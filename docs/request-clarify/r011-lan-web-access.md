# r011 局域网网页访问后端服务

日期：2026-05-15

## 需求背景

用户希望把后端服务部署在家庭局域网内的一台常开机器上，例如 NAS 或完整 Linux 主机，并让局域网内其他机器上的网页前端可以直接访问该后端 API。

## 已确认需求

| 项目 | 结论 |
| --- | --- |
| 主要访问方式 | 前端网页访问后端 API |
| 网络范围 | 家庭局域网 |
| 外部网络 | Tailscale、端口转发、反向代理暴露外网不在本次范围 |
| 部署目标 | NAS 或普通 Linux 主机均可，不做 NAS 专属优化 |
| 认证授权 | 本次先不实现认证，优先打通访问；完整认证后续单独设计 |
| 当前阻塞点 | 默认 `server.host = "127.0.0.1"` 只允许本机访问；当前代码仍使用 `[u8; 4]` 数组配置，需要改为标准 IP 字符串 |
| 配置方式 | 延续现有 `.zembra.env` TOML 配置机制 |
| Host 配置格式 | 使用标准 IP 字符串，例如 `"127.0.0.1"`、`"0.0.0.0"`、`"192.168.1.10"` |

## 本次目标

- 支持后端监听局域网网卡，使其他设备可通过 `http://<host-lan-ip>:<port>` 访问。
- 支持网页前端从局域网其他机器发起浏览器请求。
- 提供面向普通 Linux 主机的部署与访问说明。
- 保持默认开发配置偏本机安全，不把认证作为本次前置条件。

## 本次不包含

- Bearer token、账号系统、权限模型或设备授权。
- Tailscale、公网端口转发、HTTPS 证书、反向代理配置。
- NAS 平台专属打包、应用商店配置或 DSM/QNAP 等厂商适配。
- 前端工程改造，除非后续设计确认需要补充 API base URL 约定。

## 待设计点

| 事项 | 需要在设计阶段确定 |
| --- | --- |
| 监听地址示例 | 使用 `server.host = "0.0.0.0"` 作为局域网部署示例，默认开发配置保留 `"127.0.0.1"` |
| CORS 策略 | 使用允许局域网开发前端的可配置 origin，还是先采用宽松策略 |
| 配置字段 | 是否新增 `[server].cors_allowed_origins` 等字段 |
| 验证方式 | 是否通过 `/health`、`/api-docs/openapi.json` 和浏览器 CORS 请求共同验收 |
| 部署说明 | 是否新增 docs 下部署文档，或写入本需求的设计/执行计划 |

## 验收标准草案

- 在 Linux 主机上使用 `.zembra.env` 配置后，服务可以通过 `server.host = "0.0.0.0"` 监听 `0.0.0.0:<port>`。
- 局域网其他机器可以访问 `GET /health` 并得到 `200 OK`。
- 局域网网页前端可以从浏览器调用后端 API，不被 CORS 阻止。
- 本次改动不引入认证要求，不影响本机开发默认行为。
